2. Writes — operator commands by step

All write targets sit under CBT_<dev>.CTRL.CMD.* (the server prefixes CBT_ automatically). Sites in frontend/components/vfd-wizard-modal.tsx.

Step Tag (CMD.*) Type Value Site Notes
1 Verify Identity Valid_Map BOOL 1 :379 Confirms identity after F1 press.
2 HP Valid_HP BOOL 1 :562 L2 cells Motor HP (Field) + VFD HP (Field) are written first; PLC pulse is second.
3 Bump Bump BOOL 0 then 1 :681, :687 Rising-edge required (ONS on rung 7). Force 0 first so a stale latched 1 still produces an edge.
3 Polarity (batch, ~1 s loop) Reverse_Polarity + Normal_Polarity BOOL mutually exclusive 1/0 :724–725 Written via write-tags-batch in a continuous ~1 s loop so both land in the same PLC scan (rung 13 SR latch).
3 Confirm Direction Valid_Direction BOOL 1 :737 Written after the polarity pair so the routing matches before the cascade locks.
5 Calibrate Speed Run_At_30_RVS BOOL 0 then 1 :1013, :1019 Rising-edge (ONS on rung 19). PLC moves 29.99 into CommandedVelocity. Setpoint only — does not start a stopped drive; belt must already be running (e.g., via F1 keypad).
Step 4 — Controls Verified writes no PLC tag. It only persists a local row in SQLite (VfdControlsVerified) so reopening the wizard remembers the tech already verified keypad F0/F1/F2 controls. See app/api/vfd-commissioning/controls-verified/route.ts.
Step 5 — what the buttons actually do (read this carefully).
  • "Send 30 RVS Setpoint" is a speed setpoint, not a Start command. It writes one BOOL (CMD.Run_At_30_RVS) which the AOI's rung 19 turns into MOVE(29.99, CommandedVelocity). If the drive is sitting at 0 with no active run command, this changes nothing visible — VFDs need a Run command, not just a setpoint. The expected workflow is: start the belt from the VFD keypad (F1) → click Send 30 RVS Setpoint → the running belt's speed changes to 30 RVS.
  • "Save Measurement" is a record-only action. It writes the L2 cell Speed Set Up with <INITIALS> · <FPM> FPM @ <RVS> RVS. Zero PLC writes. SCADA bulk-applies these FPM/RVS pairs to drive parameters during the VFD parameter download — this is by design, not a bug.

6. Notes, quirks, and traps

Rising-edge writes need a manual reset to 0 first. CMD.Bump and CMD.Run_At_30_RVS are gated by ONS instructions on the PLC. Rung 15 fills the CMD region with 0 every scan (~10 ms), so normally the bit returns to 0 on its own — but if a previous session crashed mid-write or the scan is stalled, writing 1 on top of a latched 1 produces no edge. The wizard always writes 0 first, then 1.
The write-tag route has a Speed_FPM branch that targets STS, not CMD. If field === 'Speed_FPM' and dataType !== 'BOOL', the path becomes CBT_<dev>.CTRL.STS.Speed_FPM. The wizard does not currently invoke this branch — there is no UI write to Speed_FPM — but the code path exists. See write-tag/route.ts:53-56.
Batch route header mentions Override_RVS + RVS. Those writes are not currently issued by the wizard. Today the only batch use is the polarity pair (Reverse_Polarity + Normal_Polarity) on Step 3.
REAL writes/reads use uint32 bit reinterpretation. ffi-rs's DataType.Float is broken (returns "JsNumber can only be double type"); the codebase reads via plc_tag_get_uint32 and reinterprets, and writes via plc_tag_set_int32(floatToInt32Bits(v)). Don't switch back to native float helpers without retesting on hardware. See vfd-wizard-reader.ts:33-40, write-tag/route.ts:19-24, 105-109.
BOOL writes use plc_tag_set_int8, not plc_tag_set_bit. Matches the proven plc-client.ts approach. Switching helpers has historically broken writes silently.
Reader stays robust to missing tags. If a handle fails to create (older AOI without that bit, mis-named device, PLC unreachable), the reader retries every 3 s in the background and always broadcasts every key in the snapshot (null for missing) so the UI can show disconnected vs connected per-tag. See vfd-wizard-reader.ts:74, 187-300.
